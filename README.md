# FulcrumLib
completed test task (C#, with unit tests)

## initial description for what was expected to be done for this test task:

Develop a C# library (not Web service) that allows to create, view and edit a list of books
Each book has a title, an author, and number of pages. The library should provide following functionality:
    1. Load a list of books from an XML-file
    2. Add a new book to the list
    3. Sort the list in alphabetical order by author first
       Then for each author sort it in alphabetical order by title
       Example: first all Andersen's books, then all King's books
       Andersen's books: first The Little Mermaid, then The Ugly Duckling etc
    4. Book search by a part of its title (basic, not fuzzy)
    5. Save the list of books into an XML-file
The library should be implemented in C#, .NET
No UI is required, the library will be used by other software modules
The library should be covered with unit tests (any framework)

Коментар команди по тестовому:
ЩО СПОДОБАЛОСЬ:

2. Використання відносно сучасного синтаксису (target-typed new, scoped using, file-scoped namespace, collection initializers);

НАПРЯМКИ ПОКРАЩЕННЯ:
1. Відсутня природня валідація даних класу Book. В запропонованій реалізації можна створити книгу з пустим або null автором чи назвою, та від'ємною кількістю сторінок, що не має сенсу в предметній області.
Незважаючи на коментар "//TODO add validation (was not mentioned in task description as mandatory)" це не означає, що валідація не потрібна була.

2. Пропоную реалізувати Equals/IEquatable<Book> для класу Book. Без цього порівняння книг працює по посиланню, що робить неможливим коректну перевірку, використання Contains, Assert.Equal для об'єктів у тестах. У тестах доводиться порівнювати кожне поле окремо вручну копіпастою. Хелпер AssertBook - це обхідне рішення проблеми, яка вирішується на рівні доменної моделі.

3. Логіку сортування можна було відокремити за допомогою імплементації IComparable<Book> або надати компаратор IComparer<Book> ззовні за допомогою Dependency Inversion, щоб її можна було перевикористовувати та тестувати окремо. Зараз логіка сортування жорстко закодована всередині BookShelf.Sort().

4. Сортування змінює внутрішній список, але не повертає результат. Щоб отримати відсортований список книг клієнт повинен спочатку викликати Sort(), а потім SearchByTitle() - дві операції замість однієї. Крім того, після наступного додавання книги сортування втрачається, що може призвести до неочевидної поведінки:
shelf.Sort();
var sorted = shelf.SearchByTitle();
shelf.Add(newBook);
foreach(var book in sorted) {} // expected sorted
Варто або повертати відсортовану копію списку замість зміни внутрішнього стану, або підтримувати сортування постійно як інваріант класу (наприклад, за допомогою SortedSet\SortedList з IComparer<Book>).

5. AddBulk додає книги в циклі і викликає Add для кожного елемента. Це правильно з точки зору перевикористання валідації, але якщо IEnumerable<Book> містить null елемент посередині - частина книг буде додана, а частина ні. Немає атомарності операції. Варто або валідувати всю колекцію перед додаванням, або документувати цю поведінку.

6. Клас BookShelf порушує принцип Single Responsibility (SOLID) та змішує відповідальність за менеджмент колекції книг та сериалізацію/десериалізацію. Рекомендую передавати реалізацію читання/запису через інтерфейс ззовні (Dependency Inversion, також SOLID). Зараз BookShelf жорстко залежить від статичного класу BookPersistence, що також унеможливлює підміну для тестування через моки.

7. Немає способу отримати всі книги з окрім як через gjier - це неочевидний API. Варто додати властивість Books типу IReadOnlyList<Book> для прямого доступу до колекції.

8. Існує data race при перевірці існування файлу при десерилізації. Якщо файл видалиться в системі після перевірки File.Exists але до File.OpenRead, метод кине зовсім інше виключення, ніж мирне повернення пустого списку. Варто не перевіряти існування файлу, а обгорнути читання у try-catch і ловити FileNotFoundException, або просто віддати цей обов'язок класу роботи з файловою системою.

9. Мінорне: XmlSerializer створюється щоразу при виклику LoadFromXmlFile та SaveToXmlFile. Краще один раз створити і зберігати в статичному полі класу. В даному випадку з використаним конструктором проблем з пам'яттю не буде, але якщо хтось випадково змінить на інший конструктор XmlSerializer (з другим параметром) - це призведе до витіку пам'яті через динамічну генерацію збірок.

ТЕСТИ:
1. Хоча присутні деякі тести на помилкові сценарії, не тестуються інші граничні та помилкові випадки:
- Читання файлу невірного формату (не XML, пошкоджений XML)
- Файл коректного формату, але без книг (порожній список)
- Збереження пустої колекції
- Сортування пустої колекції
- Пошук книги, якої не існує (SearchByTitle з неіснуючою назвою)
- FillFromFile з неіснуючим файлом
- SaveToFile з невалідним шляхом
- SearchByTitle з великою/малою літерами (case-insensitive перевірка)
- FillFromFile з append (чи зберігається порядок)
тощо

2. Тести використовують конкретні реалізації (реальні файли, XML серіалізацію) замість моків. Це робить їх інтеграційними, а не юніт-тестами. Оскільки BookShelf жорстко залежить від статичного BookPersistence, використання моків неможливе без рефакторингу. Варто використовувати Dependency Inversion і моки для тестування лише бізнес-логіки без залежності від файлової системи.

3. Порівняння результатів відбувається вручну через хелпер AssertBook (кожне поле окремо, кожен індекс). Варто реалізувати Equals/IEquatable<Book> і використовувати Assert.Equal для об'єктів та колекцій напряму.

4. У BookPersistenceTests.SaveToXmlFile_PersistsDataCorrectly тест серіалізує дані тим же XmlSerializer і порівнює рядки. Це означає, що тест не перевіряє коректність серіалізації - він перевіряє що два виклики того ж серіалізатора дають однаковий результат. Якщо серіалізатор має баг - тест його не виявить. Краще десеріалізувати збережений файл і порівняти об'єкти.
